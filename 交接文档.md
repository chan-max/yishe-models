# yishe-models 交接文档

## 1. 项目定位

**独立模型推理服务器**，为 design-server 和 1s 等前端项目提供本地 AI 模型推理能力。

核心目标：**去 OpenAI 依赖**，embedding 和 LLM 推理全部本地化。

---

## 2. 为什么需要这个服务

| 痛点 | 现状 | 解决后 |
|------|------|--------|
| **embedding 依赖外部 API** | 每次向量入库都要调 OpenAI，需要 API Key + 网络 | 本地推理，零费用、零延迟 |
| **LLM 成本高** | 所有 AI 对话走外部 API（klapi / dxnt），有费用和限流风险 | 本地跑开源 LLM，按需切换 |
| **离线不可用** | 没有网络时整个向量系统瘫痪 | 完全离线可用 |
| **隐私** | 所有文本都发到第三方 | 数据不出服务器 |

---

## 3. 架构设计

```
┌──────────────────────┐     OpenAI-compatible API     ┌──────────────────────┐
│    design-server      │ ──────────────────────────▶  │    yishe-models       │
│                        │                              │                        │
│  EmbeddingService     │   POST /v1/embeddings         │  sentence-transformers │
│  (用 OpenAI SDK 调)    │   POST /v1/chat/completions   │  llama.cpp / vLLM      │
│                        │                              │  (未来)                │
│  AiService (LLM 调用) │                              │                        │
└──────────────────────┘                              └──────────────────────┘
```

**核心思路：实现 OpenAI 兼容的 API，让现有代码一行不改。**

- `EmbeddingService` 用 `new OpenAI({ baseURL: 'http://yishe-models:xxxx/v1' })` 就能直接调
- `AiService`（LLM）后续同样零改动接入

---

## 4. 技术选型

### 推荐方案：Python + FastAPI

| 组件 | 选型 | 理由 |
|------|------|------|
| **Web 框架** | FastAPI | 轻量、异步、自动生成 OpenAPI 文档 |
| **Embedding 模型** | `sentence-transformers` | 最成熟的文本 embedding 库，支持数百个预训练模型 |
| **LLM 模型** | llama.cpp + llama-cpp-python | CPU 友好，内存占用低，GGUF 量化模型生态丰富 |
| **进程管理** | systemd / pm2 | 保证服务自动重启 |

### 为什么选 Python 而不是 Node.js

- `sentence-transformers` 是 Python 生态最成熟的 embedding 库
- `@xenova/transformers`（Node.js）功能有限，模型支持少，社区小
- LLM 推理（llama.cpp/vLLM）的最佳绑定也在 Python

---

## 5. Embedding 模型选择

| 模型 | 维度 | 内存 | 质量 | 适用场景 |
|------|------|------|------|----------|
| `all-MiniLM-L6-v2` | 384 | ~300MB | ★★★ | **推荐起步**：轻量、够用 |
| `bge-small-zh-v1.5` | 512 | ~400MB | ★★★★ | 中文场景更优 |
| `multilingual-e5-large` | 1024 | ~2GB | ★★★★★ | 多语言、高质量 |
| `bge-large-zh-v1.5` | 1024 | ~2GB | ★★★★★ | 中文最佳质量 |

**推荐起步用 `all-MiniLM-L6-v2`**：
- 磁盘：~90MB
- 内存：~300MB
- 推理：50-200ms/条（CPU）
- 首次启动自动从 HuggingFace 下载，之后缓存到 `~/.cache/huggingface/`

⚠️ **关键影响**：切换 embedding 模型后，**所有 Qdrant collection 需要重建**（维度变了），已有向量数据需要 reindex。

## 6. API 设计

### 6.1 Embedding（第一期）

与 OpenAI Embedding API 完全兼容：

```
POST /v1/embeddings
Content-Type: application/json

{
  "model": "all-MiniLM-L6-v2",
  "input": "需要向量化的文本"
}
```

响应：
```json
{
  "object": "list",
  "data": [
    {
      "object": "embedding",
      "index": 0,
      "embedding": [0.01, -0.02, "...共384维"]
    }
  ],
  "model": "all-MiniLM-L6-v2",
  "usage": { "prompt_tokens": 5, "total_tokens": 5 }
}
```

### 6.2 Chat Completion（第二期）

兼容 OpenAI Chat Completion API：

```
POST /v1/chat/completions
Content-Type: application/json

{
  "model": "qwen2.5-7b-instruct",
  "messages": [
    { "role": "user", "content": "你好" }
  ],
  "temperature": 0.7,
  "max_tokens": 1024
}
```

### 6.3 健康检查

```
GET /health
→ { "status": "ok", "models": ["all-MiniLM-L6-v2"] }
```

## 7. 与 design-server 的集成

### 7.1 第一步：在 AI API Key 页面配置

在 yishe-admin → 系统管理 → AI API Key 中新增一条记录：

| 字段 | 值 |
|------|-----|
| 名称 | 本地模型服务器 |
| Base URL | `http://localhost:8900/v1` |
| API Key | `not-needed`（随便填，本地服务不校验） |
| 默认模型 | `all-MiniLM-L6-v2` |
| 公开 | 是 |
| 启用 | 是 |

### 7.2 第二步：修改 collection 维度

`design-server/src/vector-search/collections/collection-configs.ts`

把 6 个 collection 的 `dimensions: 1536` 全部改为 `dimensions: 384`

### 7.3 第三步：重建 Qdrant collection + reindex

1. 重建所有 Qdrant collection（新维度 384）
2. 调用 `POST /sticker/reindex-vectors` 重新索引现有贴纸
3. 其他模块（knowledge、fonts、sentences）按需 reindex

### 7.4 EmbeddingService 改动量

**零改动。** 因为 EmbeddingService 使用 OpenAI SDK，只要 AI API Key 中的 `baseUrl` 指向模型服务器即可。

## 10. 性能预估

以 `all-MiniLM-L6-v2` 为例，在普通服务器（4 核 CPU、8GB RAM）上：

| 操作 | 耗时 |
|------|------|
| 模型加载（启动） | 3-5 秒 |
| 单条 embedding | 50-200ms |
| 批量 100 条 embedding | 2-5 秒 |
| 常驻内存 | ~300MB |
